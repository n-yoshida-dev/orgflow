# フェーズ1専用学習対象定義

この文書は、Phase 1 で「何を学ぶか」「何を作るか」「まだ優先しないものは何か」を明確にする。
Phase 1 の完了に必要なバックエンドの基盤を先に固め、学習テーマが広がりすぎることを防ぐ。

---

## 0. この文書の位置づけ

Phase 1 は学習ロードマップ全体の土台を作るフェーズである。
ここで作るべきものは機能数が多いアプリではなく、React・AWS・Terraform・Go への置き換えに進む前提になる、説明可能なバックエンドの基盤である。

---

## 1. Phase 1 のゴール

Java / Spring Boot で OrgFlow API v1 を作り、「B2B ワークフロー SaaS のバックエンドを、設計理由まで含めて説明できる状態」にすること。

完了条件:

- 組織、ユーザー、ロール、申請、承認、監査ログが、それぞれ何を表す概念か説明できる
- API がログイン済みの利用者だけを受け付ける状態になっている
- ロールごとに実行できる操作が変わる
- 申請作成 → 承認または差し戻し → 監査ログ記録の流れが壊れずに動く
- OpenAPI が API 契約として管理されている
- 単体テストと統合テストがあり、CI で自動実行に成功する
- README と ADR を読めば、なぜその設計にしたかを説明できる

---

## 2. Phase 1 で扱う題材

B2B ワークフロー SaaS 風の「OrgFlow」。申請・承認・監査・権限・組織切替・データ分離を含む業務 SaaS。

### 最低限扱う業務概念

- `organizations`: 利用者や申請データが属する組織単位
- `users`: システムを利用する人
- `roles`: 利用者が実行できる操作の権限区分
- `requests`: 利用者が作成する申請データ
- `approvals`: 申請に対する承認・差し戻し・却下などの判断記録
- `audit_logs`: 誰が、どの組織で、いつ、どの操作を行ったかの証跡

### 最低限扱う業務フロー

ログイン → 組織切替 → 申請作成 → 承認 → 差し戻し → 監査ログ記録

---

## 3. Phase 1 で学習対象に含めるもの

### 3-1. ドメイン理解

コードを書く前に、利用者・ロール・申請・承認・差し戻し・監査ログ・組織切替の意味を整理する。先にテーブルやDTOを作ってから意味づけする進め方は採らない。

### 3-2. DB設計

業務概念の意味が崩れないこと、後から大きく壊れにくいことを優先する。エンティティ分割、主キー/外部キー、一意制約、状態遷移、監査/履歴、テナント境界を扱う。

### 3-3. API設計

業務上の操作として説明できること、後から実装言語を変えても契約が保てることを優先する。リソース設計、request/response、エラーコード使い分け、OpenAPI契約を扱う。

### 3-4. Java / Spring Boot 実装

Controller / Service / Repository の役割分担、DTO と Entity の境界、認証・認可の判定位置、業務フローを壊さない実装、例外処理、監査ログ記録、更新競合への備えを扱う。「どの処理をどの層に置いたか」を説明できることを求める。

### 3-5. 認証・認可

ログイン済み API の成立、認証と認可の分離、ロール別の操作制御、403 の適切な返却を扱う。認証方式そのものの比較検討は ADR で扱う。

### 3-6. RBAC

Viewer / Approver / Admin などのロール差分、ロール別の操作制御、API側でのアクセス制御を扱う。全パターン網羅の複雑な権限マトリクスまでは求めないが、雑な設計では不十分。

### 3-7. 監査ログ

誰が・どの組織で・いつ・何をしたか・状態変更の内容を記録する。後付けの補助機能ではなく、最初から業務フローの一部として設計・実装する。

### 3-8. テスト

単体テスト、統合テスト、主要業務フロー検証、権限差分検証、監査ログ検証、409 Conflict 検証を扱う。カバレッジの数字より、壊してはいけない業務フローを守れていることを優先する。

### 3-9. CI

build、test、CI の成功状態の維持、push / PR のたびに品質確認が走ることを扱う。CD や本番デプロイ自動化は主対象ではない。

### 3-10. OpenAPI

API 一覧、request / response、主要なエラー、将来の React / Go 進行時にも維持したい契約として扱う。

### 3-11. README / ADR

README: 起動方法、テスト方法、主要機能、技術選定の概要。
ADR: 題材選定理由、認証方式選定理由、監査ログを最初から含める理由、OpenAPI を先に置く理由。

### 3-12. ドキュメントの位置づけ

コードと同等以上に文書を重視する。README、ADR、OpenAPI、テスト、CI、学習ログは成果物の一部とみなす。README は採用側やレビュアーが全体像を理解できる文書、ADR は面接で語るための材料として扱う。

---

## 4. Phase 1 でまだ主対象にしないもの

将来は必要でも、Phase 1 の完了を遅らせてまで先にやるものではない。

- React / TypeScript の実務品質 UI
- UI の見た目の改善（デザイン、ダークモード、トースト等）
- Go 実装（Phase 4）
- AWS 本番相当構成（Phase 3）
- Terraform の本格運用
- Kubernetes
- 個社特化技術（Rails, Kotlin, GCP, 検索基盤, LLM基盤, OpenTelemetry等）
- AI 機能をアプリの中心にすること

---

## 5. Phase 1 の学習順序

1. 題材理解とユースケース整理
2. ドメイン整理
3. DB設計
4. API の責務整理
5. OpenAPI の骨子作成
6. Spring Boot で API 実装
7. 認証・認可 / RBAC の組み込み
8. 監査ログ実装
9. 単体テスト / 統合テスト
10. Docker Compose での再現
11. GitHub Actions CI
12. README / ADR 更新

理由: ドメイン・契約・データ構造を先に固めないと、認可・監査・テスト追加時に設計が崩れやすい。

---

## 6. Phase 1 のレビュー重点観点

- ユースケース理解が浅くないか
- DB設計が業務概念と整合しているか
- 状態遷移が業務フローとして無理がないか
- 認証と認可を混同していないか
- RBAC が API 側で成立しているか
- 監査ログが後付けになっていないか
- OpenAPI が契約として機能しているか
- テストが壊してはいけない流れを守れているか
- README / ADR に判断理由が残っているか

---

## 7. Phase 1 の不合格パターン

- CRUD はあるが、申請→承認→監査ログの流れが弱い
- ロールによる権限差分がない
- ログイン済み API として成立していない
- OpenAPI 契約が曖昧
- テストがほとんどない / CI がない
- README が起動手順だけで設計理由がない / ADR がない
- UI や周辺技術に時間を使い、バックエンドの基盤が弱い

---

## 8. Phase 1 完了判定

すべて満たしたときに完了とする。

- `docker compose up` で `api` / `db` が起動する
- ログイン、申請作成、承認、監査ログ記録が API で通る
- RBAC により、ロールごとに実行できる操作が変わる
- 単体テスト / 統合テストが CI で成功する
- OpenAPI と README が更新されている
- ADR が最低 3 本ある

---

## 9. Phase 1 で守る運用ルール

- Must を先に満たす / Should・Could は遅れたら削れる状態にしておく
- 設計理由を残しながら進める / 毎回 Why と採用しなかった案を記録する
- 学んだ内容は 1 本のアプリに統合する
- 自分で説明できない生成物を成果物にしない
- 学習テーマを広げすぎない
- Done 定義を満たしてから次のフェーズへ進む
