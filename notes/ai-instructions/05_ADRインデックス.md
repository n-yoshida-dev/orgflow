# ADRインデックス（Phase 1）

この文書は、Phase 1 で残すべき ADR（Architecture Decision Record＝設計判断の記録）の一覧を示す。
目的は、「なぜその設計を採ったのか」を後から説明できるようにし、README・面接・振り返りで再利用できる状態を作ること。

---

## 0. ADR運用ルール

- Phase 1 の間に最低 3 本は必ず残す（理想は 4〜6 本）
- 採用した案だけでなく、採用しなかった案とその理由も書く
- 「今はこの案を採るが、将来どの条件で見直すか」まで書く
- 1 本の ADR では 1 つの判断だけを扱う
- 長く書きすぎることより、判断理由が明確であることを優先する
- README から辿れるようにしておく
- 判断した時点で書く（後でまとめて書こうとしない）

---

## 1. 必須候補の ADR

### ADR-001: なぜ題材をタスク管理ではなく申請・承認にしたか

比較する案: ToDo アプリ / タスク管理アプリ / 社内申請・承認アプリ

決めること: ToDo やタスク管理では不足しやすい論点は何か、申請・承認で何を扱いやすくなるか、権限・監査・ワークフロー・組織切替との結びつき、転職用成果物としての説明しやすさ。

### ADR-002: 認証方式の選定理由

比較する案: セッションベース認証 / JWT ベース認証 / 外部 IdP 前提の設計

決めること: Phase 1 で優先すべきは認証済み API の成立か認証基盤の深掘りか、学習コストと説明可能性のバランス、React・AWS・Go 置換への接続性。

### ADR-003: 監査ログを最初から設計対象に含める理由

比較する案: アプリケーションログだけ / 監査用テーブルを独立保持 / イベント履歴として明示保持

決めること: 後から追加すると何が崩れやすいか、業務 SaaS で監査が重要な理由、記録の粒度と範囲、運用手順書や可観測性との接続。

### ADR-004: OpenAPI を先に置く理由

比較する案: 実装先行で後から仕様整備 / 最初に骨子だけ / 最初から API 契約として管理

決めること: 実装先行の問題点、React 側との接続、Go 置換時の維持性、契約固定の意味、学習速度との兼ね合い。

## 2. 追加候補の ADR

### ADR-005: RBAC の最小方針

決めること: 最低限どのロールを置くか、API 側制御を主にする理由、画面非表示だけでは不十分な理由、将来の権限マトリクスへの接続。

### ADR-006: 業務フローの状態遷移をどう表現するか

決めること: requests と approvals の分け方、status カラム中心か履歴テーブル分離か、不正な状態遷移の防止箇所。

### ADR-007: 409 Conflict をどう扱うか

決めること: 楽観ロックを入れるか、最小限の 409 だけ扱うか、扱わない場合の理由、Phase 2 への持ち越し条件。

### ADR-008: Docker Compose をローカル再現性の基準にする理由

決めること: 手動起動だけでは弱い理由、api と db を揃える意味、Phase 1 で AWS まで進まない理由、README・CI との接続。

## 3. ADR の最小テンプレート

```
# ADR-XXX タイトル

## ステータス
提案中 / 採用 / 保留 / 廃止

## 日付
YYYY-MM-DD

## 背景
何を決める必要があったか。

## 比較した案
候補 A / 候補 B / 候補 C

## 採用した案
今回採用した案。

## 採用理由
なぜその案を選んだか。

## 採用しなかった案と理由
他の案を採らなかった理由。

## この判断で得られるもの
何が良くなるか。

## この判断で受け入れる制約
何をあきらめるか、何を将来へ持ち越すか。

## 見直す条件
どの条件で再検討するか。
```

- ADR-001: 題材選定理由
- ADR-002: 認証方式の選定理由
- ADR-003 または ADR-004 のいずれか

理想は 4 本以上。
