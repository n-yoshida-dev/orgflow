# レビュー観点チェックリスト

この文書は、AI がレビュー時に確認すべき観点を整理したチェックリストである。
レビュー依頼時に参照する。ただし出力見出しの固定テンプレートではない。
対象テーマと依頼内容に応じて優先順位を付け、重複する指摘は統合してよい。

---

## 0. レビューの基本姿勢

- いきなり完成形に書き換えない
- まず、対象の良い点または成立している点を確認する
- そのうえで、主要な懸念点を優先して指摘する
- 最後に、次に考えるべき問い、または次の1手を示す
- 可能な限り、レビューを優先し、修正版の提示は最小限にする
- 用語の意味が曖昧な場合は、設計の良し悪しを論じる前に、言い換えと定義の修正を優先する

---

## 1. 全テーマ共通

### 1-1. 要件理解

- 何を実現したいのかが明確か
- 誰が使う機能かが明確か
- ユースケースが曖昧なまま設計に入っていないか
- 正常系だけでなく異常系も想定されているか

### 1-2. ドメイン理解

- 業務上の意味に沿ったモデルになっているか
- 状態・責務・権限の違いが曖昧になっていないか
- 一見単純なCRUDに落とし込みすぎていないか

### 1-3. 将来変更耐性

- 将来の仕様追加に耐えられるか
- 拡張時に責務が破綻しないか
- 変更点が局所化される構造になっているか

### 1-4. 説明可能性

- なぜその設計・実装にしたか説明できるか
- 他案との比較ができるか
- README / ADR に残すべき判断点があるか

### 1-5. 説明の分かりやすさ

- 造語を定義なしで使っていないか
- 抽象語を使う場合、このアプリで何を指すか言い換えているか
- README や ADR にそのまま書いても意味が通るか

---

## 2. DB設計

- このテーブルは何の業務概念を表しているか明確か
- 別概念を1テーブルに詰め込んでいないか / 分けすぎていないか
- 主キー、外部キー、一意制約、NOT NULL / CHECK 制約は妥当か
- 冗長なデータや更新不整合のリスクはないか
- ステータス管理が曖昧でないか / 不正な状態遷移を防ぐ考えがあるか
- 作成者、更新者、日時、承認履歴の保持は必要か
- テナント境界を意識した構造か
- 物理削除か論理削除か、その理由はあるか
- 後で index が必要になりそうな検索条件は何か

---

## 3. API設計

- このAPIが何をするか一文で言えるか / 複数責務を詰め込んでいないか
- リソース指向として自然か / 動詞過多やネスト過剰でないか
- 誰が呼べるか明確か / 認証と認可が分けて考えられているか
- request / response の責務が適切か / 内部情報を漏らしていないか
- 400 / 401 / 403 / 404 / 409 / 422 / 500 の使い分けを考えているか
- 同じリクエストの再送に耐えられるか / 競合を意識しているか
- OpenAPI で表現しやすい設計か

---

## 4. コード

- Controller / Service / Repository / Domain の責務が混ざっていないか
- クラス名・メソッド名・変数名が役割を表しているか
- 一読で処理の流れが追えるか / ネストや条件分岐が複雑すぎないか
- 例外を握り潰していないか / HTTP層に返すエラーと内部例外が整理されているか
- DB更新、ログ出力等の副作用が見えやすいか
- null / 空 / 不正値 / 境界値への考慮があるか
- 将来の要件追加でどこが壊れそうか

---

## 5. テスト

- 何を保証するテストか明確か / 正常系だけで満足していないか
- 単体 / 統合 / E2E の切り分けが適切か
- 正常系、異常系、境界値、権限差分を押さえているか
- テストが実装詳細に依存しすぎていないか / 壊れやすくないか
- 監査ログや権限制御の確認が必要か
- ワークフローの状態遷移を押さえているか

---

## 6. 認証 / 認可

- 誰であるかを確認する仕組みが明確か
- 何ができるかの判定が明確か / ロール・権限・所属組織の関係が曖昧でないか
- 将来的なRBAC実装に接続しやすいか
- 誰が何を実行したか追跡できるか / 認可失敗時の扱いも含めて設計できているか

---

## 7. README / ADR

### README

- 何を作ったかが一目でわかるか
- セットアップ手順が再現可能か
- 技術選定理由が簡潔に説明されているか
- 採用担当 / レビュアーが読みやすいか

### ADR

- 何を決めた文書か明確か
- 選択肢比較があるか / なぜその案にしたかが書かれているか
- 将来の見直しポイントがあるか

---

## 8. インフラ / CI

- Docker: 開発環境の再現性があるか / 責務が分かれているか
- GitHub Actions: 最低限の品質ゲートになっているか / 壊れたとき原因が追いやすいか
- Terraform: 今のフェーズで過剰設計になっていないか
