# domain-and-rules
## このアプリでは何を扱うか
本アプリでは**経費申請アプリ**を題材とする。
本題材を通し、申請・承認・監査ログ・認証・認可・認可ポリシー・組織などの概念を実装し、学習することが目的である。

## 登場人物と権限
本アプリで登場する登場人物や権限についての用語を以下に一覧化する。
### 登場人物(概念)
- 申請（request）
- 承認（approval）
- 承認ポリシー（approval policy）
- 組織（organization）
- 役割（role）
- 監査ログ（audit log）

### 権限
- 申請権限（request authority）
- 承認権限（approval authority）
- 管理者権限（admin authority）


## 概念定義
以下では各登場人物の概念定義を記載する。

## request とは何か
### 何を表す概念か
requestは申請内容および、申請の現在状態の正本をを持つ。

### ライフサイクル
requestは作成されてから承認済み・却下・取り消し・差し戻しまで継続する。
※差し戻し後の再申請時は新版が新規作成される。

### 件数
requestは新規申請直後は第1版として作成され、差し戻し後の再申請の度、新版として新規requestが作成される。

### その他
requestの現在状態は1つのみしかない。
requestの現在状態は申請中 / 承認済み / 差し戻し / 却下 / 取消の5つからなる。
1つの申請には1人の申請者がいる。
requestは必ずいずれかの組織に属する。
requestは現在の承認段の情報を持ち、多段承認を管理する。これにより現在状態が「申請中」の場合でも、現在の承認段と組み合わせることで1次承認待ちなのか2次承認待ちなのか、といった承認段を判断する。
却下済みの申請に対し、申請内容の編集はできない。
却下済みの申請に対し、再承認はできない。
差し戻し後に再申請を行う場合、編集後のrequestはバージョンが上がり、新版として新規requestとして管理される。
再申請後の新規requestは、大本のrequestの管理IDとバージョン管理IDの組み合わせで管理する（例：初版 管理ID12345、版ID1、第二版 管理ID12345、版ID2）。
requestは現在の承認段を情報として持つ。
requestは申請時点のポリシーを基にして判断された承認担当情報（承認可能な候補集合など）や適用済み承認ルートを参照できる。

## approval とは何か
### 何を表す概念か
approval は request に対して起きる判断（いつ誰がどのような判断を行ったか）の履歴である。

### ライフサイクル
単体のapproval自体は発生した時点で完了し、継続しない。
requestが却下された場合、後続の承認段は進行せず、以降の新たなapprovalは発生しない。
なお、承認段の管理はrequestにて行う。

### 件数
approvalは1つの申請に対して0回以上発生する（申請が提出後すぐに取り消されるケースは承認が0回となる可能性がある）。

### その他
approvalは必ず既存の申請(申請中ステータス)に対して行われる。
対象のrequestは、requestの管理IDと版IDの組み合わせで紐づける。
承認者はその時点での承認権限を持っていなければならない。
同じ承認段階で複数の承認者がいた場合、承認の重複は起こらない（誰かが承認すると、ルートに従い次の段の承認フェーズとなる）。
次の承認者は誰か、現在何段目の承認かといった情報ははapprovalでは持たない。


## approval policy とは何か
### 何を表す概念か
approval policyとは、所属組織・申請種別・申請内容による承認ルートの差を定めたポリシーである。

### 申請後のポリシー変更
申請後にポリシーが変更となった場合でも、ポリシーが変更される前に行われた申請には変更前のポリシーが固定され、次回の申請から変更後のポリシーが適用される。

### ポリシーの条件例
以下の違いにより、承認ルートや承認段数が変化する。
・申請が属する組織（例：部付の社員は直接部長が承認を行う）
・申請の種別（例：接待費は経理部の承認も必要とする）
・申請の内容（例：金額が10万円より低い場合は部長承認はスキップする）

## 認可の条件
特定の申請に対して承認できる条件は以下である。
・承認者に申請が属している組織の承認権限が付与されていること
・承認者がその request の現在承認段の対象であること
・承認者がrequestの承認ルート上にあること

## user / organization / role の関係は、認可判定のためにどこまで必要か
organizationはデータ境界の基準となる。
roleは操作権限であり、同一userでもorganization毎に操作権限が異なる。例えばAユーザーはX組織では管理者であるが、Y組織では一般申請者になる。
なお、承認ルートはapproval policyにより定められる。

## audit logとは何か
### 何を表す概念か
audit logとは、申請・承認の操作だけでなく、ユーザー追加/削除/変更やポリシー変更、システムログインなど、監査・追跡に必要な重要操作の証跡を保管する。

### audit logの種類
例として、以下のような内容がaudit logに含まれる。
・操作種別
・操作ユーザー
・操作ユーザーの所属組織
・操作対象
・操作日時

# 各関係の多重度と文脈
## user と organization
多重度：多対多
文脈：userは組織ごとに異なる操作権限を持つことができる。そのため、多対多となる。

## user と role
user と role は直接結ばない。user の organization 所属を介して role が付与される。

## request と approval
多重度：1対多
文脈：1requestに対し承認操作が発生する度にapprovalが発生するため、1対多となる。

## request と approval policy
文脈：requestの発生時にその時点でのapproval policyを参照し、approval policyによる承認ルートなどがrequestに適用される

## audit log は各概念への外部参照として持つのか
文脈：audit logは各監査ログ毎にrequest / approval / policy / user などの対象識別子を持ち、各概念と監査ログを紐づけることができる。
