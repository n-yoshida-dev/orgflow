## **概要**

本アプリは、**経費申請アプリ**を題材とした B2B ワークフロー SaaS 風アプリである。
この題材を通して、申請、承認、監査ログ、認証、認可、承認ポリシー、組織といった業務システムに特有の概念を整理し、実装可能な形に落とし込むことを目的とする。

本書では、アプリが扱う業務ルール、不変条件、状態遷移、および未確定論点を整理する。

---

## **登場人物**

本アプリで扱う登場人物は以下のとおりである。

- `user`
  アプリを利用する主体である。

- `requester`
  申請を作成・再申請・取消できる利用者である。

- `approver`
  承認対象の申請に対して、承認・差し戻し・却下を行う利用者である。

- `admin`
  組織、ユーザー、ロール、承認ポリシーなどの管理操作を行う利用者である。

---

## **業務概念**

本アプリで扱う主要な業務概念は以下のとおりである。

- `request`
  申請内容と、その申請版の現在状態を表す業務データである。

- `approval`
  request に対して行われた判断の履歴である。

- `approval policy`
  所属組織、申請種別、申請内容に応じて、どのような承認ルートを適用するかを決めるルールである。

- `organization`
  user が所属する単位であり、データ閲覧境界および権限適用範囲の基準である。

- `role`
  organization への所属に対して付与される操作権限の区分である。

- `audit log`
  重要操作の証跡を記録する監査用ログである。

---

## **権限**

本アプリでは、少なくとも以下の権限を扱う。

- 申請権限

- 承認権限

- 管理者権限

これらの権限は user に直接付与されるのではなく、**organization への所属を介して role として適用される**。

---

## **request に関する業務ルール**

### **request の意味**

1つの `request` は、**1つの申請版**を表す。
申請内容そのものと、その申請版の現在状態を保持する正本である。

### **request の状態**

request の現在状態は常に1つであり、以下のいずれかを取る。

- 申請中

- 承認済み

- 差し戻し

- 却下

- 取消

### **request の版管理**

新規申請時には、第1版の request が作成される。
差し戻し後に再申請する場合は、既存 request を上書きせず、**新版の request を新規作成する**。

同一の申請系列は、たとえば以下のように管理する。

- 申請系列 ID

- 版番号

例:

- 申請系列 ID: 12345, 版番号: 1

- 申請系列 ID: 12345, 版番号: 2

### **request の所属と申請者**

- 1つの request には、1人の申請者が存在する。

- request は必ず1つの organization に属する。

### **request が保持・参照する情報**

- request は現在の承認段を保持する。

- request は、申請時点で決定された承認担当情報および適用済み承認ルートを参照できる。

### **request に関する不変条件**

- 却下済み request は編集できない。

- 却下済み request は再承認できない。

- 現在状態が「申請中」であっても、現在承認段と組み合わせることで、何段目の承認待ちかを判定できる。

## **approval に関する業務ルール**

### **approval の意味**

`approval` は、request に対して行われた判断の履歴である。
誰が、いつ、どのような判断を行ったかを記録する。

### **approval のライフサイクル**

1つの approval は、発生した時点で完了する。
継続的な状態は持たない。

また、request が却下された場合、後続の承認段は進行せず、それ以降の新たな approval は発生しない。

### **approval の件数**

1つの request に対して、approval は 0 回以上発生する。
たとえば、申請直後に request が取り消された場合、approval が 0 件のまま終了することがある。

### **approval に関する不変条件**

- approval は必ず既存の request に対して行われる。

- approval の対象 request は、申請系列 ID と版番号の組み合わせで特定される。

- approval は「申請中」の request に対してのみ行われる。

- 承認者は、その時点で承認権限を持っていなければならない。

- 同一承認段に複数の承認者がいる場合、承認は重複しない。誰か1人が承認した時点で、ルートに従って次段へ進行する。

- approval 自体は、次の承認者や現在承認段といった進行管理情報は持たない。進行管理は request 側で行う。

### **承認操作が許可される条件**

特定の request に対して承認操作が許可されるのは、少なくとも以下の条件を満たす場合である。

- 承認者に、その request が属する organization における承認権限が付与されていること

- 承認者が、その request の現在承認段の対象者であること

- 承認者が、その request に適用された承認ルート上の承認対象者であること

---

## **approval policy に関する業務ルール**

### **approval policy の意味**

`approval policy` は、所属組織、申請種別、申請内容に応じて、承認ルートや承認段数を決定するルールである。

### **policy の適用タイミング**

request 作成時には、その時点の approval policy を用いて承認ルートを決定する。
申請後に approval policy が変更された場合でも、**既存の request には申請時点のルールを適用した結果を維持する**。変更後の policy は、次回以降の新規 request に適用される。

### **policy 条件の例**

承認ルートや承認段数は、以下の条件によって変化しうる。

- request が属する organization

- 申請種別

- 申請内容

- 金額条件

例:

- 部付の社員は、部長承認から開始する

- 接待費は、経理部承認も必要とする

- 金額が 10 万円未満の場合は、部長承認をスキップする

---

## **organization と role に関する業務ルール**

### **organization**

organization は、user が所属する単位である。
また、organization はデータ閲覧境界および権限適用範囲の基準となる。

user は、1つ以上の organization に所属できる。

### **role**

role は操作権限の区分であり、user に直接付与されるのではなく、**organization への所属に対して付与される**。

同一 user であっても、organization ごとに role が異なりうる。
たとえば、ある user は organization X では admin、organization Y では requester であることがある。

---

## **audit log に関する業務ルール**

### **audit log の意味**

`audit log` は、申請・承認だけでなく、ユーザー追加、ユーザー変更、ポリシー変更、ログインなど、監査・追跡に必要な重要操作の証跡を記録する。

### **audit log に含める代表項目**

- 操作種別

- 操作ユーザー

- 操作ユーザーの所属 organization

- 操作対象

- 操作日時

audit log は、request / approval / approval policy / user などの対象識別子を保持し、各概念との追跡可能性を確保する。
