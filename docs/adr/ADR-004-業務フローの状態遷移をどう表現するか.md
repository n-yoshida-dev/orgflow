# ADR-004: 業務フローの状態遷移をどう表現するか

## ステータス
Accepted

## 日付
2026-03-19

## コンテキスト
経費申請アプリでは、申請、承認、差し戻し、却下、取消を扱う。

このとき、`request` の現在状態、`approval` の履歴、再申請時の版管理をどう持つかを決める必要がある。

本アプリでは、`request` は申請版の現在状態を持つ正本、`approval` は判断履歴として扱いたい。また、差し戻し後の再申請では、旧版を上書きせず新版を作成したい。

## 選択肢

### Option 1: request 1つに現在状態も承認履歴も詰め込む
- メリット: 一見単純
- デメリット: 履歴が汚れやすく、責務が混ざる

### Option 2: request と approval を分離し、再申請時は既存 request を上書きする
- メリット: 一部は分離できる
- デメリット: 差し戻し前後の版差分が追いにくい

### Option 3: request と approval を分離し、再申請時は新版 request を作る
- メリット: 現在状態、履歴、版差分を分離できる
- デメリット: キー設計と参照が少し複雑になる

## 決定
**request と approval を分離する。**

`request` は 1 つの申請版の現在状態を持つ正本とする。

`approval` はその request 版に対する判断履歴とする。

差し戻し後の再申請では、既存 request を上書きせず、**同一申請系列の新版 request を新規作成する**。

## 採用理由
現在状態と履歴を分けることで、責務が明確になる。

また、再申請時に新版を作ることで、差し戻し前後の差分、approval の帰属先、audit log の追跡先を明確にできる。

承認ワークフローの現在位置は request 側で管理し、approval 側は進行管理を持たない方が自然である。

## 不採用理由
request に履歴も持たせる案は責務が混ざる。

上書き再申請は、旧版との差分説明や監査で弱い。

## トレードオフ
request と approval の参照が一段複雑になる。

再申請時の版管理も必要になる。

その代わり、業務フローを自然に説明できる。

## 将来の見直し条件
将来、より厳密な履歴比較やドラフト保存が必要になったら、申請系列と版管理の扱いを再評価する。

## 実装方針
- request は現在状態を持つ
- approval は履歴のみを持つ
- request は現在承認段を持つ
- request 作成時に適用済み承認ルートを確定する
- request は `申請系列 ID + 版番号` で一意に識別できるようにする
