# ADR-004: 業務フローの状態遷移をどう表現するか

## ステータス

Accepted

## 日付

2026-03-19

## コンテキスト

経費申請アプリでは、申請、承認、差し戻し、却下、取消を扱う。

このとき、次の3点をどう持つかを決める必要がある。

- `request` の現在状態
- `approval` の履歴
- 差し戻し後に再申請したときの版管理

本アプリでは、`request` は**その申請版について現在有効な申請データ**として扱い、`approval` は判断の履歴として扱いたい。  
また、差し戻し後の再申請では、旧版を上書きせず、新版を作成したい。

## 選択肢

### Option 1: `request` 1つに現在状態も承認履歴もまとめて持つ

- メリット: 一見単純
- デメリット: 現在状態と履歴が混ざる

### Option 2: `request` と `approval` を分離し、再申請時は既存 `request` を上書きする

- メリット: 一部は分離できる
- デメリット: 差し戻し前後の差分を追いにくい

### Option 3: `request` と `approval` を分離し、再申請時は新版 `request` を作る

- メリット: 現在状態、履歴、版差分を分けて扱える
- デメリット: キー設計と参照が少し複雑になる

## 決定

`request` と `approval` を分離する。

`request` は、**1つの申請版について現在有効な申請データ**とする。

`approval` は、その `request` 版に対して行われた判断の記録とする。

差し戻し後の再申請では、既存の `request` を上書きせず、**同じ申請系列の新版 `request` を新しく作成する**。

## 採用理由

現在状態と履歴を分けることで、各概念が何を表すのかが明確になる。

また、再申請時に新版を作ることで、次のことを説明しやすくなる。

- 差し戻し前後で、どこが変わったか
- どの `approval` がどの申請版に対応するか
- どの `audit_log` がどの申請版に対応するか

さらに、「現在何段目の承認待ちか」という進行中の情報は `request` 側で持ち、`approval` は過去の判断記録だけにする方が役割分担が明確である。

## 不採用理由

`request` に履歴まで持たせる案は、現在状態と履歴が混ざる。

上書き再申請は、差し戻し前後の比較や監査の説明がしにくい。

## トレードオフ

`request` と `approval` の参照関係は少し複雑になる。

また、再申請時の版管理も必要になる。

その代わり、申請の現在状態、過去の判断履歴、差し戻し後の再申請をそれぞれ分けて説明できる。

## 将来の見直し条件

将来、より厳密な版比較やドラフト保存が必要になったら、申請系列と版管理の持ち方を見直す。

## 実装方針

- `request` は現在状態を持つ
- `approval` は履歴だけを持つ
- `request` は現在承認段を持つ
- `request` 作成時に、適用する承認ルートを確定する
- `request` は `申請系列ID + 版番号` の組み合わせで一意に識別できるようにする

## 追記: request のキー設計（2026-03-24）

request は request_id（サロゲートキー）を主キーとして持つ。
申請系列内での版の一意性は、(request_series_id, version_no) の複合一意制約で表現する。

理由: approvals など他テーブルからの外部キー参照を複合キーで行うと JOIN が複雑になるため、単一のサロゲートキーを参照に使い、業務キーは制約として持つ。
