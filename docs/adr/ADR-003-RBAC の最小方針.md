# ADR-003: RBAC の最小方針

## ステータス
Accepted

## 日付
2026-03-19

## コンテキスト
Phase 1 では、organization / role / request / approval / audit log を扱う。

このとき、役職、承認ルート、操作権限を混ぜると設計が崩れる。

本アプリでは、`role` は操作権限、`approval policy` は承認ルート決定ルールとして扱いたい。また、role は user に直接付与するのではなく、organization への所属を介して適用したい。

## 選択肢

### Option 1: 役職をそのまま認可根拠にする
- メリット: 一見わかりやすい
- デメリット: 兼務、代理、将来の例外対応に弱い

### Option 2: role に承認ルート条件まで持たせる
- メリット: テーブルは少なく見える
- デメリット: 操作権限と業務ルールが混ざる

### Option 3: role と approval policy を分離する
- メリット: 操作権限とルート決定を別責務で扱える
- デメリット: 概念数は増える

## 決定
Phase 1 の RBAC は、**organization 所属を介した role 付与**を最小方針とする。

また、**role は操作権限のみを表し、承認段数や金額条件は approval policy 側で扱う**。

認可根拠に役職は使わない。

## 採用理由
同一 user でも organization ごとに権限が変わりうる。

また、承認対象者の決定は「その request の現在承認段と適用済み承認ルート」で決まるべきであり、role だけで決めるべきではない。

これにより、権限境界と業務ルールの境界を分けられる。

## 不採用理由
役職を認可根拠にすると、兼務、代理、将来の例外対応に弱い。

role に承認段数条件まで持たせると、意味が肥大化する。

## トレードオフ
概念が 1 つ増えて、初見の理解は少し難しくなる。

その代わり、後から organization、policy、route を拡張しても崩れにくい。

## 将来の見直し条件
閲覧権限や代理承認を追加する段階で、role の粒度と policy 側の条件表現を再評価する。

## 実装方針
最小 role は以下とする。

- requester
- approver
- admin

認可は UI だけでなく API 側で強制し、対象 request の organization 所属と role を確認する。

さらに、承認操作では現在承認段と適用済み承認ルートも確認する。
