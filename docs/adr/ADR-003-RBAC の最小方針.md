# ADR-003: RBAC の最小方針

## ステータス

Accepted

## 日付

2026-03-19

## コンテキスト

Phase 1 では、`organization` / `role` / `request` / `approval` / `audit_log` を扱う。

このとき、次の3つを混ぜると設計が崩れやすい。

- 役職
- 承認ルート
- 操作権限

本アプリでは、`role` は「何の操作ができるか」を表し、`approval_policy` は「どの条件でどの承認ルートを適用するか」を表すものとして分けて扱いたい。

また、`role` は `user` に直接付与するのではなく、**どの組織に所属しているか**という情報を介して適用したい。

## 選択肢

### Option 1: 役職をそのまま認可の根拠にする

- メリット: 一見わかりやすい
- デメリット: 兼務、代理、将来の例外対応に弱い

### Option 2: `role` に承認ルートの条件まで持たせる

- メリット: テーブル数は少なく見える
- デメリット: 操作権限と業務ルールが混ざる

### Option 3: `role` と `approval_policy` を分ける

- メリット: 操作権限と承認ルート決定ルールを別々に扱える
- デメリット: 概念数が増える

## 決定

Phase 1 の RBAC は、**組織への所属情報に対して `role` を付与する**方針とする。

また、`role` は操作権限だけを表し、承認段数や金額条件は `approval_policy` 側で扱う。

認可の根拠に役職は使わない。

## 採用理由

同じ `user` でも、所属する `organization` ごとに権限が変わりうる。

また、承認対象者の決定は、`role` だけではなく、次の条件で決まるべきである。

- その `request` が今どの承認段にいるか
- その `request` にどの承認ルートが適用されているか

このため、「何の操作ができるか」を表す `role` と、「どの順で誰が承認するか」を表す `approval_policy` は分けた方がよい。

## 不採用理由

役職をそのまま認可の根拠にすると、異動、兼務、代理対応で設計が崩れやすい。

`role` に承認段数や金額条件まで持たせると、意味が広がりすぎて説明しにくくなる。

## トレードオフ

概念が1つ増えるため、最初は理解しにくくなる。

その代わり、後から `organization`、`approval_policy`、承認ルートの条件を拡張しやすくなる。

## 将来の見直し条件

閲覧権限や代理承認を追加する段階で、`role` の分け方と `approval_policy` の条件表現を見直す。

## 実装方針

最小の `role` は次の3つとする。

- `requester`
- `approver`
- `admin`

認可は UI だけでなく API 側でも強制する。

API では少なくとも次を確認する。

- 対象 `request` が属する `organization` に、その利用者が所属していること
- 必要な `role` を持っていること
- 承認操作であれば、その `request` の現在承認段と適用済み承認ルートに照らして承認対象者であること

## 追記: organization_membership と role の関係（2026-03-24）

role は organization_membership に直接持たず、membership_roles という中間テーブルを介して付与する。

理由: 同じ組織への所属に対して複数のロールが付きうるため、organization_memberships テーブルに role_id カラムを直接持つと1ロールしか表現できない。

membership_roles は (membership_id, role_id) の組み合わせを1件として持つ。
