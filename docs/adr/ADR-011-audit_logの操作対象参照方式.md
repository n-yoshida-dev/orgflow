# ADR-011: audit_logの操作対象参照方式

- Status: Accepted
- Date: 2026-03-25

## 背景

- `audit_log` は `request`・`approval`・`user`・`approval_policy` など
  複数の種別の概念を横断して操作を記録する必要がある
- 各テーブルへの外部キー制約を持つ案と、
  対象種別と対象識別子を文字列・UUID で持つ案を検討した
- 決めないと、参照先レコードが削除されたときに監査記録まで失われるリスクがある

## 採用した案

- `target_type varchar`（操作対象の種別名、例: "request" / "user" / "approval_policy"）と
  `target_id uuid`（操作対象の識別子）の2カラムで持つ
- `target_id` に対する外部キー制約は置かない

## 採用理由

- 監査ログは操作が行われた時点の不変な証跡であり、
  参照先が後から削除されても記録を消してはいけない
- 外部キー制約を置くと、参照先レコードの削除時に制約エラーが発生するか、
  `ON DELETE SET NULL` にすると誰に対する操作だったかの情報が失われる
- `target_type` と `target_id` の2カラムにすることで、
  「この request に対して行われた操作を全件取得する」というクエリを
  `WHERE target_type = 'request' AND target_id = ?` の形で書ける

## 不採用案

- 各対象テーブルへの外部キー制約を持つ案:
  参照先削除時に制約エラーが発生するか、NULL になって追跡不能になる
- 対象種別と識別子を1カラム（varchar）にまとめる案（例: "request:uuid"）:
  WHERE 句での検索が文字列パターンマッチになり、インデックスが効きにくい

## 受け入れる制約

- `target_id` の参照整合性は DB が保証しない
- `target_type` の値の正規化はアプリコードで担保する必要がある

## 見直す条件

- 監査ログの検索要件が複雑化し、対象種別ごとに専用テーブルを持つ方が
  クエリ効率で有利になった場合
