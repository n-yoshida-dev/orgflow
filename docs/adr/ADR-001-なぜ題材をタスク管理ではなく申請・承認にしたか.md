# ADR-001: なぜ題材をタスク管理ではなく申請・承認にしたか

## ステータス
Accepted

## 日付
2026-03-19

## コンテキスト
Phase 1 では、Java / Spring Boot による REST API を中心に、組織、権限、監査、状態遷移を含む業務アプリの芯を実装したい。

単純な Todo や個人タスク管理では CRUD は学べるが、B2B SaaS らしい論点である「承認フロー」「組織境界」「RBAC」「監査ログ」を自然に含めにくい。

転職用ポートフォリオとしても、「単純な CRUD を作った」ではなく、「業務ルールを設計し、それを API とデータモデルに落とした」と説明できる題材が必要だった。

## 選択肢

### Option 1: Todo アプリ
- メリット: 学習コストが低い
- デメリット: 状態遷移、権限、監査の重みが弱い

### Option 2: タスク管理アプリ
- メリット: 複数ユーザーやステータスを入れやすい
- デメリット: 承認や監査を自然に入れにくい

### Option 3: 申請・承認アプリ
- メリット: 承認、差し戻し、却下、組織、RBAC、監査を自然に含められる
- デメリット: 画面の派手さは弱い

## 決定
Phase 1 の題材として、**経費申請を中心とした申請・承認アプリ**を採用する。

## 採用理由
この題材なら、申請 → 承認 → 監査という業務フローを中心に据えられる。

また、organization / role / request / approval / audit log を分離しながら設計する必要があり、単純 CRUD では見えない設計判断を README や ADR に残せる。

将来的にも、React フロント、Go バックエンド、通知、監査画面、CSV 出力などへ拡張しやすい。

## 不採用理由
Todo とタスク管理は、Phase 1 で扱いたい「業務 SaaS らしい制約」を自然に表現しにくい。

今回の優先は UI 映えではなく、バックエンドの芯と説明可能性である。

## トレードオフ
見た目のわかりやすさは下がる。

その代わり、ドメイン理解、状態遷移、認可、監査という中核論点を扱える。

## 将来の見直し条件
Phase 2 以降で UI 訴求を強めたい場合は、申請・承認アプリの上にダッシュボードや通知機能を積む。

題材自体は変更しない。

## 実装方針
Phase 1 では、経費申請の最小フローとして以下を成立させる。

- request 作成
- 承認 / 差し戻し / 却下
- audit log 記録
- organization / role に基づく API 認可
