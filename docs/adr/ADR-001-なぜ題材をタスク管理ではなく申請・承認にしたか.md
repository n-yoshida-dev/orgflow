# ADR-001: なぜ題材をタスク管理ではなく申請・承認にしたか

## ステータス

Accepted

## 日付

2026-03-19

## コンテキスト

Phase 1 では、Java / Spring Boot による REST API を中心に、組織、権限、監査、状態遷移を含む業務アプリとして最低限成立する部分を実装したい。

単純な Todo アプリや個人向けタスク管理アプリでも CRUD 自体は学べる。
しかし、それらの題材では、B2B SaaS で重要になりやすい次の論点を無理なく入れにくい。

- 承認フロー
- 組織ごとのデータ分離
- RBAC
- 監査ログ

転職用ポートフォリオとしても、「単純な CRUD を作った」ではなく、「業務ルールを整理し、それを API とデータモデルに落とした」と説明できる題材が必要だった。

## 選択肢

### Option 1: Todo アプリ

- メリット: 学習コストが低い
- デメリット: 状態遷移、権限、監査の論点が弱い

### Option 2: タスク管理アプリ

- メリット: 複数ユーザーやステータスを入れやすい
- デメリット: 承認や監査を題材として無理なく入れにくい

### Option 3: 申請・承認アプリ

- メリット: 承認、差し戻し、却下、組織、RBAC、監査を自然に扱える
- デメリット: 見た目の派手さは出しにくい

## 決定

Phase 1 の題材として、**経費申請を中心とした申請・承認アプリ**を採用する。

## 採用理由

この題材であれば、申請 → 承認 → 監査という業務フローを中心に据えられる。

また、`organization` / `role` / `request` / `approval` / `audit_log` を分けて設計する必要があり、単純な CRUD では見えにくい設計判断を README や ADR に残せる。

将来的にも、React フロントエンド、Go バックエンド、通知、監査画面、CSV 出力などへ拡張しやすい。

## 不採用理由

Todo アプリとタスク管理アプリは、Phase 1 で扱いたい「組織」「権限」「監査」「状態遷移」を題材として十分に表現しにくい。

今回の優先は、見た目のわかりやすさではなく、サーバーサイド設計と説明可能性である。

## トレードオフ

画面だけを見ると、何のアプリかは少し伝わりにくくなる。

その代わり、次の論点を1つの題材で扱える。

- 業務ルールの整理
- 状態遷移
- 認可
- 監査

## 将来の見直し条件

Phase 2 以降で UI 面の訴求を強めたい場合は、申請・承認アプリの上にダッシュボードや通知機能を追加する。

題材自体は変更しない。

## 実装方針

Phase 1 では、経費申請の最小フローとして次を成立させる。

- `request` の作成
- 承認 / 差し戻し / 却下
- `audit_log` の記録
- `organization` と `role` に基づく API 認可
