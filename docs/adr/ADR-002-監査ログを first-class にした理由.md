# ADR-002: 監査ログを first-class にした理由

## ステータス
Accepted

## 日付
2026-03-19

## コンテキスト
経費申請アプリでは、承認履歴だけでなく、ログイン、ユーザー変更、ロール変更、ポリシー変更などの重要操作も追跡したい。

`approval` は判断履歴であって、監査証跡の代替ではない。

承認だけを追えればよいのではなく、誰が、どの organization で、いつ、何をしたかを後から確認できる必要がある。

## 選択肢

### Option 1: アプリケーションログだけ残す
- メリット: 実装が軽い
- デメリット: 業務上の追跡性が弱い

### Option 2: approval だけで履歴を追う
- メリット: テーブル数は増えない
- デメリット: 承認以外の重要操作を追えない

### Option 3: audit log を独立概念として持つ
- メリット: 横断的に重要操作を追跡できる
- デメリット: 設計対象が 1 つ増える

## 決定
`audit log` を、**重要操作の証跡を横断的に記録する first-class の独立概念**として扱う。

## 採用理由
監査対象は承認だけではない。

誰が、どの organization で、いつ、何をしたかを追跡できるようにするには、approval と別の audit log が必要である。

業務 SaaS らしい制約や説明可能性を README / ADR / 面接で示すうえでも、監査を後付けにしない方がよい。

## 不採用理由
アプリケーションログだけでは、業務上の監査証跡として弱い。

approval だけでは、ログインや設定変更を表現できない。

## トレードオフ
保存対象が増える。

どこまでを監査対象にするかの線引きも必要になる。

その代わり、追跡可能性と説明責任が大きく向上する。

## 将来の見直し条件
将来、イベント駆動化する場合は、audit log とイベント履歴の関係を再整理する。

Phase 1 ではそこまでは扱わない。

## 実装方針
最低限、以下を記録する。

- 操作種別
- 操作ユーザー
- 操作ユーザーの所属 organization
- 操作対象種別
- 操作対象識別子
- 操作日時
