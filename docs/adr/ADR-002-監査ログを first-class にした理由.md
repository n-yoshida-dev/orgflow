# ADR-002: requestとapprovalの関係性

## ステータス
Accepted

## 日付
2026-03-17

## コンテキスト（背景・課題）
approvalはrequestに対して行われた判断の履歴であるが、これをrequest内で持つかどうかの検討があった。
ER図作成の判断材料とするため、不変ルールや概念同士の関係の方針決定が必要であった。

## 決定（Decision）
- requestとapprovalを分離し、approvalは承認判断の履歴とする。
- requestは承認ルートと現在の承認段を参照させることで、承認状態を紐づける。

## 根拠（Rationale）
- approvalを分離することで、承認履歴を追いやすくなる利点があった。
- requestは申請内容と現在の申請ステータス、版管理IDなどのみを管理することで責務を分解することができる。
- requestに情報を詰め込みすぎると設計が破綻する可能性があった。
- requestの新規発生・更新のタイミングとapprovalの発生タイミングは異なるため、同一レコードで管理すると不要な情報まで増加し、データ肥大の可能性があった。

## 検討した選択肢（Options）
### Option 1: request に履歴も現在状態も全部載せる
- request内で「第一承認」「第二承認」「第・・・」のようにカラムを持ち、承認がある毎に内容を上書きする。
- メリット
  - データを一元管理できる
- デメリット
  - 例えば第二承認の段で「差し戻し→再申請→承認」といった流れがあった場合、差し戻しの履歴が上書きされ、流れが追えなくなってしまう。
  - 承認は順番に発生するため、各承認毎の発生時刻などの付加情報も管理する必要があり、カラムが肥大化する。

### Option 2: 状態管理を approval 側に寄せる
- 概要
- メリット
- デメリット

### Option 3: request と approval を分ける
- 概要
- メリット
- デメリット

## 結果（Consequences）
### 良い影響
- 責務が分解され、データとしてわかりやすくなる。
- approvalの履歴が正確に終える。

### 悪い影響
- 特になし
