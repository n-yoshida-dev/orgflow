# ADR-002: 監査ログを他の概念の付属情報ではなく独立して設計する理由

## ステータス

Accepted

## 日付

2026-03-19

## コンテキスト

経費申請アプリでは、承認履歴だけでなく、ログイン、ユーザー変更、ロール変更、ポリシー変更などの重要操作も後から確認したい。

`approval` は承認・差し戻し・却下の判断記録であって、監査記録そのものではない。

必要なのは、承認履歴だけではない。  
**誰が、どの組織で、いつ、何をしたか**を、申請以外の操作も含めて確認できる必要がある。

## 選択肢

### Option 1: アプリケーションログだけを残す

- メリット: 実装が軽い
- デメリット: 業務上の監査記録としては不十分

### Option 2: `approval` だけで履歴を追う

- メリット: テーブル数は増えない
- デメリット: 承認以外の重要操作を記録できない

### Option 3: `audit_log` を独立した概念として持つ

- メリット: 重要操作を共通の形式で追跡できる
- デメリット: 設計対象が1つ増える

## 決定

`audit_log` は、**重要操作を後から追跡するための独立した記録**として扱う。

## 採用理由

監査対象は承認だけではない。

次のような操作も記録対象に含めたい。

- ログイン
- ユーザー変更
- ロール変更
- 承認ルール変更
- 申請作成
- 承認 / 差し戻し / 却下

これらを一貫した形で追跡するには、`approval` とは別に `audit_log` を持つ必要がある。

また、業務SaaSらしい設計判断として、監査を後付けではなく最初から設計対象に含める方が説明しやすい。

## 不採用理由

アプリケーションログだけでは、業務上必要な監査記録としては弱い。

`approval` だけでは、ログインや設定変更を表現できない。

## トレードオフ

保存対象が増えるため、設計と実装は少し重くなる。

また、どの操作まで監査対象にするかを決める必要がある。

その代わり、後から操作履歴を確認できる範囲が大きく広がる。

## 将来の見直し条件

将来、イベント駆動の仕組みを入れる場合は、`audit_log` とイベント履歴の役割分担を再整理する。

Phase 1 ではそこまでは扱わない。

## 実装方針

最低限、次の情報を記録する。

- 操作種別
- 操作ユーザー
- 操作ユーザーの所属 organization
- 操作対象種別
- 操作対象識別子
- 操作日時
